1) WHAT THIS PROJECT IS

This is no longer just a simple Excel viewer. It is a client-facing BI microsite built with Next.js 14 + TypeScript that:

- Loads a workbook by default from a generated JSON file at build/dev time.
- Still supports fallback Excel upload in the browser through an Admin modal.
- Converts Excel sheets into an executive dashboard, audit/data-quality views, raw sheet exploration, methodology pages, duplicates review, funnel metrics, and ASCVD cardiovascular risk views.
- Includes additional standalone routes for population reporting, satisfaction/NPS, and chart pages driven by Excel-derived JSON.
- Is designed to run without a backend. Data stays client-side after the JSON is generated.

The app has evolved into a hybrid between:

- An executive review portal.
- A static-exportable data microsite for cPanel.
- A small analytics platform with multiple report routes.

5) HOW THE EXCEL CONVERTER WORKS

File:
- scripts/convert-xlsx-to-json.ts

Main responsibilities:

- Reads source workbook from data/source.xlsx
- Handles merged cells
- Detects the first non-empty header row
- Trims and de-duplicates headers
- Preserves all rows
- Converts each sheet to an array of row objects
- Adds source_sheet to every row
- Adds poblacion_norm for KIO rows when possible
- Generates inferred.semanticMap using lib/semantic.ts
- Optionally extracts clinical chart tables from data/Graficas_Con_Formulas_y_Tablas.xlsx
- Converts addon clinical workbooks in data/addons into JSON files

Generated workbook.json structure:

- meta
  generatedAt, sourceFileName, chartsFileName, sheetNames, rowCounts

- sheets
  Object keyed by sheet name, each containing an array of row objects

- excelCharts
  Extracted chart-ready blocks for clinical conditions when detected

- inferred
  semanticMap detected from BASE_MODELADA

Important note:
The converter preserves raw business data and adds support fields rather than replacing the originals.

7) SEMANTIC DETECTION

File:
- lib/semantic.ts

Current detected semantic fields:

- cliente
- jornada
- poblacion
- sexo
- rangoEdad
- personId
- riskCategory

Detection is fuzzy and based on normalized header matching.

Examples:

- cliente matches cliente / company / idcliente / id_client
- jornada matches jornada / fecha / date / periodo
- poblacion matches poblacion / population / cohorte
- sexo matches sexo / genero / gender / sex

This is lightweight and practical, but still narrow. If the next project uses very different header naming, this layer will need to be expanded.


8) DATA NORMALIZATION

File:
- lib/model.ts

Current transformations:

- Trims all string values
- Normalizes sexo values
  Examples:
  M, HOMBRE, MASCULINO -> MASCULINO
  F, MUJER, FEMENINO -> FEMENINO

- Derives poblacion_norm for KIO:
  If value contains IT -> IT
  If value contains DATA -> DATA
  Else -> SIN_CLASIFICAR

Special behavior:

- There is a hardcoded override list of some names that are forced from SIN_CLASIFICAR to IT.
  This is a very project-specific business rule tied to the current KIO dataset and should be reviewed before reusing the project for another client.


9) CORE BI / SELECTOR LOGIC

File:
- lib/selectors.ts

This file contains most of the business logic.

Key implemented behavior:

9.1 Filters

- Builds filter configuration from BASE_MODELADA using detected semantic columns
- Supports:
  Cliente
  Jornada
  Poblacion
  Sexo
  Rango_Edad

- For KIO population filters, if poblacion_norm exists it uses:
  IT -> IT Services
  DATA -> Data Centers
  SIN_CLASIFICAR -> Sin clasificar

9.2 Generic distributions

- buildDistribution()
- buildDistributionWithDenominator()
- buildStackedByGroup()

Charts and KPI percentages are based on non-empty applicable rows for the respective field.

9.3 Condition registry

The project implements canonical clinical condition mapping:

- Diabetes
- Prediabetes
- HTA
- Pre-HTA
- Sobrepeso
- Obesidad
- Tabaquismo
- Dislipidemia

It prefers *_Flag columns and falls back to *_Cat columns, while excluding helper columns such as Aplicable, Fuente, raw, DemoValido, TieneLabs, etc.

Per-row condition derivation:

- true if any matching source column is positive
- false if there is usable data and it is negative
- null if all matching source columns are empty

This is used by:

- condition filters
- condition KPI cards
- applicable denominators
- OR vs AND comorbidity filtering

9.4 Applicable sample logic

The current portal already distinguishes:

- Base universe = rows after normal filters
- Applicable universe = rows where the selected condition(s) are not null
- Cases = rows where selected condition(s) are true

This behavior is implemented mainly through:

- lib/kpi.ts

Important nuance:
Condition-specific applicability is generic and derived from the registry columns. It is not yet a fully formal clinical contract per condition beyond available columns.

9.5 KIO population reconciliation

Implemented logic:

- Todos = total filtered rows
- IT = rows with poblacion_norm === IT
- Data = rows with poblacion_norm === DATA
- Sin clasificar = rows with poblacion_norm === SIN_CLASIFICAR
- Delta = Todos - (IT + Data + Sin)

This appears in the dashboard as a visible reconciliation block.

9.6 Funnel detection

The funnel logic is heuristic:

- Searches for columns that look like inscritos / inscripcion
- Searches for clinico / muestra / completo / valid
- Searches for finalizado / terminado / completado

Then computes:

- Total RAW / Total registrados
- Inscritos
- Muestra clinica valida
- Finalizaron

This is useful, but fragile if the next dataset uses different field names or workflow states.

9.7 ASCVD detection

The app looks for:

- ASCVD percentage column
- ASCVD category column

If found, it computes:

- applicable sample
- high risk percentage for >= 20 when pct exists
- category distribution when category exists

No imputation is done.

9.8 Data quality

- summarizeQuality() scans numeric columns in DATA_QUALITY
- tries to detect checked / missing / duplicates totals based on header keywords

9.9 Duplicates

- If personId exists, duplicates are grouped by that field
- Otherwise, duplicates are grouped by full-row hash

Records are flagged, not removed.

21) DEPLOYMENT / HOSTING STATE

Current next.config.js:

- output: "export"
- trailingSlash: true
- basePath: "/vitalbit/kio-review"
- assetPrefix: same as basePath
- NEXT_PUBLIC_BASE_PATH injected from next.config.js

This means the project is currently configured for static export and subdirectory hosting.

Important reality:

- middleware.ts exists and implements Basic Auth for Next server deployments
- BUT when using static export on cPanel, middleware does not protect the site
- In cPanel/static hosting, access protection must be handled via Apache .htaccess / .htpasswd

So there are two different auth models in the codebase:

- Next middleware auth for server deployments
- Apache Basic Auth for static export deployments

That is a real architectural tension to keep in mind for the next iteration.

23) CURRENT LIMITATIONS / RISKS / DEBT

23.1 Hardcoded client logic

The app is still partially specialized for KIO/ORIFLAME.

Examples:

- KIO population normalization
- hardcoded override names in lib/model.ts
- route-specific labels referencing KIO DATA and KIO IT
- chart pages built around the current clinical workbook structure

23.2 Semantic detection is still narrow

If the next dataset uses different headers, field detection may fail silently or degrade too much.

23.3 Funnel logic is heuristic

The operational funnel depends on column-name guessing, not a formal contract.

23.4 README is partially outdated

README still talks about Vercel and DASHBOARD_PASSWORD, while next.config.js is currently tuned for static export under a specific basePath.

23.5 Two competing deployment modes

The repo currently mixes:

- server-style middleware auth
- static-export cPanel deployment

This should probably be rationalized in the larger project.

23.6 basePath is client-specific

The current basePath is:

- /vitalbit/kio-review

That is deployment-specific and would need to become configurable if the next project has multiple environments or clients.

23.7 Addon/chart extraction is custom

Clinical chart extraction from Graficas_Con_Formulas_y_Tablas.xlsx depends on specific row/column patterns and may not generalize well.

25) WHAT SHOULD BE REFACTORED FIRST FOR A BIGGER PRODUCT

If this project is going to scale to other clients and other data models, the first refactors should be:

- Separate domain logic from presentation components
- Convert semantic/header detection into configurable contracts per dataset
- Move KIO-specific rules into a client adapter or dataset profile
- Formalize a data contract for:
  people identifiers
  client/segment/population
  clinical flags/categories
  operational funnel states
  geography fields
  survey metrics

- Normalize deployment mode:
  either real Next server hosting
  or static export only

- Make basePath configurable by env instead of hardcoding a client path
- Add validation reporting during data:build so missing critical fields are reported clearly
- Potentially split the project into:
  core analytics shell
  client configuration package
  dataset mapping/config


26) HOW I WOULD DESCRIBE THE CURRENT PRODUCT TO ANOTHER ENGINEER

This project is a static-exportable Next.js BI portal that transforms one or more Excel workbooks into a browser-based executive review experience. It already contains a reusable analytics shell, client-side state management, auto-loading of generated workbook JSON, clinical condition filtering with applicable denominators, raw sheet exploration, data quality auditing, duplicate surfacing, operational funnel reporting, ASCVD risk analysis, and extra report routes for population, satisfaction, and extracted Excel chart views. It is usable today, but its business logic is still partly tied to the current KIO/ORIFLAME clinical dashboards and would need modularization before being generalized to larger multi-client datasets.
